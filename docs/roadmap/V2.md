# Zaide Roadmap V2 — IDE Core Upgrade

**Status:** Planned

**Phase range:** Phase 8 through Phase 13

**Current implementation phase:** None — Phase 8 requires its own verified
implementation plan before coding begins.

Roadmap V1 delivered the IDE shell, agent workspace foundation, terminal, and
local Git workflow. Roadmap V2 turns that foundation into an IDE that can carry
the everyday edit, understand, build, run, test, and debug loop.

```text
V1 = Workspace Shell + Initial Agent/Git Foundation
V2 = IDE Core
V3 = AI-Native Orchestration
```

V3 is expected to deepen the existing agent foundation with memory, browser and
tool use, and automation. V2 deliberately focuses on making the standalone IDE
core reliable first.

---

## Roadmap Decisions

- Phase numbering remains global: V2 continues at Phase 8.
- C# is the first language-intelligence and debugging target.
- Linux is the primary V2 validation platform. New architecture should avoid
  unnecessary platform coupling, but V2 does not promise equal validation on
  macOS and Windows.
- Command registration and keybinding resolution precede Command Palette work.
- Minimal workspace/project discovery precedes LSP integration; the full
  Build/Run/Test workflow follows it.
- Output, Problems, and Test results are structured IDE surfaces, not aliases
  for the interactive PTY terminal.
- Document formatting is part of the C# LSP phase.
- Multi-cursor editing is deferred beyond V2. It must not expand Phase 9.
- Exact LSP server, DAP adapter, protocol libraries, secret-store mechanism,
  and performance budgets are implementation-plan decisions, not roadmap
  assumptions. Each must be verified against the live stack before adoption.

## Phase 8 — Core Platform and Settings

**Goal:** Remove user-facing hardcoding and establish the shared infrastructure
needed by later V2 features.

### Scope

- Versioned application settings schema
- `SettingsService` with explicit load, save, validation, and defaults
- Durable settings persistence
- Settings UI
- Editor defaults such as font, size, whitespace, and indentation preferences
- LLM endpoint, provider, and model configuration for the existing execution path
- Secret-handling boundary for credentials
- Environment-variable fallback for the existing agent configuration
- Command registry with stable command identifiers
- Default keybindings, user overrides, and conflict handling
- Minimal workspace/project identity and discovery seam required by later phases

### Boundaries

- API keys must not be written as plaintext into the ordinary settings file.
- This phase does not build a broad multi-provider agent platform.
- This phase does not implement Command Palette UI, LSP, build execution, or
  debugging.
- Structured Townhall or activity-history persistence is not implied by the
  application-settings store.

### Exit Direction

Later phases can consume settings and commands without adding new hardcoded
keybindings or reading provider values directly from process environment state.

---

## Phase 9 — Editor UX

**Goal:** Make everyday editing faster and more predictable.

### Scope

- Command Palette backed by the Phase 8 command registry
- User-configurable editor commands and shortcuts
- Search and Replace
- Code folding
- Tab navigation, ordering, closing, and dirty-state UX improvements
- Editor and status-bar feedback improvements
- Consistent focus, caret, and selection behavior across editor commands

### Boundaries

- Multi-cursor editing is deferred beyond V2.
- Language-semantic completion, diagnostics, navigation, and formatting belong
  to Phase 10.
- Phase 9 must reuse the existing document/tab ownership model unless a verified
  blocking defect requires a separately documented structural change.

### Exit Direction

Common editing actions are discoverable through the Command Palette, bindable
through settings, and usable without introducing language-server dependencies.

---

## Phase 10 — C# Language Intelligence

**Goal:** Turn the editor into a C#-aware IDE surface through the Language
Server Protocol.

### Scope

- LSP client and language-server process lifecycle
- Initialize, shutdown, cancellation, restart, and failure handling
- Workspace and document synchronization
- Diagnostics and a truthful Problems projection
- Completion
- Hover information
- Go to Definition
- Document and workspace symbol information
- Document Formatting through LSP `textDocument/formatting`
- Atomic application of returned text edits
- Formatting integrated with undo, dirty state, caret, and selection behavior
- `Format Document` command registered through the Phase 8 command system
- Configurable formatting shortcut, initially `Ctrl+Shift+I`
- Optional Format on Save setting, disabled by default

### Boundaries

- C# is the only required language target in V2.
- Range Formatting and Format on Type are not required for Phase 10 exit.
- Zaide does not implement a separate formatter engine when the selected C#
  language server can provide document formatting.
- The selected server and client library must be proven in the Phase 10
  implementation plan before feature milestones begin.
- A failed, cancelled, timed-out, or unsupported formatting request must leave
  the document unchanged.

### Exit Direction

A user can open a C# workspace and receive diagnostics, completion, hover,
definition navigation, symbols, and safe whole-document formatting.

---

## Phase 11 — Project Workflow

**Goal:** Support the normal build loop without leaving Zaide.

### Scope

- Complete solution and project detection for the supported C# workflow
- Explicit Build, Run, and Test commands
- Configurable execution profiles where the workflow requires them
- Cancellation and one-operation-at-a-time policy
- Structured Output panel
- Problems integration for parsed build failures
- Test-results surface
- Navigation from build, test, and diagnostic errors to source locations
- Clear separation between structured process output and interactive terminal
  sessions

### Boundaries

- V2 does not require a general-purpose task-runner ecosystem.
- Project commands must not silently execute merely because a folder opens.
- Debugging belongs to Phase 12.

### Exit Direction

A user can detect a supported C# project, build it, run it, execute tests, read
structured results, cancel work, and navigate actionable failures.

---

## Phase 12 — C# Debugging

**Goal:** Add the first complete interactive debugging path through the Debug
Adapter Protocol.

### Scope

- DAP client and debug-adapter lifecycle
- Launch configuration for the supported C# workflow
- Breakpoint creation, removal, persistence, and visual state
- Start, continue, pause, stop, step over, step into, and step out
- Current execution location
- Threads and call stack
- Scopes and variables
- Debug console/output integration
- Truthful handling of adapter startup, disconnect, crash, and protocol errors

### Boundaries

- C# on Linux is the required V2 validation target.
- Attach-to-process, remote debugging, compound sessions, data breakpoints, and
  multi-language debugging are not required for V2.
- The adapter and transport must pass a lifecycle proof in the Phase 12
  implementation plan before UI milestones begin.

### Exit Direction

A user can launch a supported C# project, stop at a breakpoint, inspect the call
stack and variables, step through code, and end the session cleanly.

---

## Phase 13 — Release Hardening

**Goal:** Make the V2 feature set stable, measurable, recoverable, and ready for
a release closeout.

### Scope

- Measured startup, editor, large-file, LSP, build, test, and debug performance
- Performance fixes against budgets locked in the Phase 13 plan
- Error handling and recovery across settings, LSP, processes, and DAP
- Settings-schema migration and rollback behavior
- Process and resource cleanup
- End-to-end tests for critical V2 workflows
- Linux release smoke matrix and documented validation status for other platforms
- Accessibility and keyboard-only workflow audit for new V2 surfaces
- Documentation, architecture, library catalog, and roadmap truth-sync

### Boundaries

- Phase 13 stabilizes V2; it does not introduce V3 agent orchestration features.
- “Performance optimization” is not an open-ended goal. The Phase 13 plan must
  define measurable budgets and representative fixtures before changes begin.

### Exit Direction

All V2 exit conditions are verified, known limitations are documented, critical
workflows have automated and manual evidence, and the roadmap can be closed
without aspirational completion claims.

## Dependency Order

```text
Phase 8: Settings + Commands + Project Foundation
    ↓
Phase 9: Editor UX
    ↓
Phase 10: C# LSP + Formatting
    ↓
Phase 11: Build + Run + Test
    ↓
Phase 12: C# DAP Debugging
    ↓
Phase 13: Release Hardening
```

Phases are intentionally sequential at roadmap level. A phase may be split into
smaller versioned sub-phases after live-code inspection, but later-phase scope
must not be pulled into an earlier implementation plan for convenience.

---

## V2 Non-Goals

- Multi-cursor editing
- Multi-language LSP or debugging parity
- Remote or collaborative workspaces
- Plugin marketplace or general plugin SDK
- Cloud settings synchronization
- Full multi-provider agent orchestration
- Agent memory, browser control, generalized tool use, or autonomous workflows
- Git push, pull, merge, rebase, or remote-host integration unless scheduled by
  a later roadmap revision

These may be reconsidered in V3 or a separately approved follow-up. Their
presence here does not reserve implementation architecture in V2.

---

## Roadmap-Level Exit Conditions

- [ ] Phase 8 through Phase 13 each have a live-code-verified implementation plan.
- [ ] Each phase is completed and closed independently before the next phase is
  described as active.
- [ ] Existing V1 editor, terminal, agent, Townhall, and Git workflows remain
  regression-covered.
- [ ] The C# edit → understand → build → run/test → debug loop works on the
  supported Linux validation environment.
- [ ] New commands are discoverable and keyboard configurable.
- [ ] Credentials are not persisted in the ordinary plaintext settings file.
- [ ] Documentation reports exact automated and manual verification evidence.
- [ ] Deferred items and platform limitations remain explicit.
