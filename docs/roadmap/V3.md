## 1. Version Direction

```text
V1 = Workspace Shell + Initial Agent/Git Foundation
V2 = IDE Core
V3 = AI-Native Orchestration
```

V1 established the Townhall workspace, direct agent panels, mechanical
`@mention` routing, terminal, editor, and local Git workflow. V2 made Zaide a
credible standalone IDE for the C# edit, understand, build, run, test, and
debug loop.

V3 must make agents capable of performing real work through that IDE while
keeping their actions understandable, reviewable, permission-controlled, and
replaceable. A successful V3 is not an IDE with a more elaborate chat panel.
It is an environment where human and agent participants can share
conversations, context, tools, and work products without hiding ownership or
execution details.

The standalone IDE remains valuable without any agent backend. V3 must extend
the V2 IDE core rather than turning it into a thin shell around one provider or
one proprietary harness.

## 3. Locked Product Principles

The following principles are accepted V3 direction unless later evidence
forces an explicit roadmap decision to revise them.

1. **Townhall is the shared conversation workspace.** Public channels and
   direct conversations belong in one coherent navigation and conversation
   model.
2. **Agent direct conversations are not a second-class side panel.** The
   current dedicated agent-panel surface should be absorbed into Townhall only
   after behavior and lifecycle parity are proven.
3. **Discord is an information-architecture reference, not Zaide's visual
   identity.** Familiar channel/DM navigation may be reused, but the current
   Discord-like appearance is temporary. Design evolves alongside features
   and through explicitly bounded visual refactors.
4. **Native and external agents receive equal product treatment.** Zaide may
   report truthful capability differences, but must not label ACP-backed
   agents as inferior or hide them behind a degraded conversation experience.
5. **Zaide builds its own native harness.** The harness is research-driven and
   optimized for real agent performance, not for novelty or code ownership
   purity.
6. **Compatible open-source code may be copied or adapted.** License
   compliance, copyright notices, provenance, and modifications are mandatory.
7. **ACP is independent of the native harness.** It is an alternative agent
   backend path for users who prefer another harness or a non-API execution
   form. ACP is not a wrapper around the Zaide native harness.
8. **Zaide is not an API product.** Internal contracts and protocol adapters
   exist to support the application; V3 does not commit Zaide to selling or
   operating a public agent API.
9. **Agent work is observable and controllable.** File edits, commands, tool
   calls, permissions, results, failures, and cancellation must remain
   reviewable.
10. **Transparency is layered.** Normal UI shows useful structured activity;
    users may opt into raw input/output and transport traces when the backend
    exposes them.
11. **Context quality matters more than context volume.** IDE events and
    memory are selected, budgeted, and attributable rather than dumped into
    every prompt.
12. **Future human-to-human conversations must not be made impossible.** V3
    does not need to deliver User-to-User messaging, but the participant model
    must not hard-code every direct conversation as User-to-Agent.
13. **Conversation visibility is explicit.** Every message and event has one
    owning `ConversationId` and a visibility scope. Direct-conversation content
    is never copied into a public channel implicitly. Cross-posted summaries
    require an explicit target plus policy or user action.

### 3.1 Decision status map

The roadmap uses three decision states. **Locked direction** is product law for
later M0 planning, but still does not authorize implementation. **Recommended**
is a preferred starting point that its owning M0 must verify. **Open** must not
be selected by assumption.

| Decision | Status | Owning verification |
|----------|--------|---------------------|
| Townhall owns public channels and direct conversations | Locked direction | Refactor 7, Refactor 8, Phase 14 M0 |
| Native Harness is first-party and research-driven | Locked direction | Later harness research/implementation M0 |
| ACP is an independent, equally presented backend | Locked direction | Later ACP implementation M0 |
| Open-source reuse requires license and provenance compliance | Locked direction | Every reuse decision and release gate |
| Conversation visibility is explicit; no implicit DM-to-public mirroring | Locked direction; **M4 shipped** | Phase 14 M4 |
| One durable Agent Identity spans sessions and runtimes | Recommended | Refactor 7 M0 |
| One initial Agent Session attaches to one Conversation | Recommended | Refactor 7 M0 |
| Context levels and policy precedence in §10 | Recommended | Owning context feature M0 |
| Exact source/module tree and assembly boundaries | Open | Refactor 6.1 M0 |
| Persistence engine, ACP version, provider list, and final visual design | Open | Their later implementation M0s |

## 4. Live Baseline and Observed Structural Debt

The following facts describe the checkout at V2 closeout and must be rechecked
by each relevant M0.

### 4.1 Agent and Townhall baseline

- `AgentExecutionService` sends one non-streaming OpenAI-compatible
  `/chat/completions` request containing only the current user message.
- There is no provider registry, tool calling, retry loop, native harness,
  ACP backend, conversation-history submission, or session-resume contract.
- All current agent panels use the same effective global endpoint, API key,
  and model configuration. Panel identity is not an execution identity.
- Current `@mention` routing resolves one visible agent name and dispatches the
  stripped user text to another panel. It does not transfer source-agent
  context, delegation lineage, or a result back to a parent agent run.
- Agent-panel output is `ObservableCollection<string>` on the direct conversation
  projection. Panel send writes user/assistant/error entries to the panel's
  `ConversationId` only (Phase 14 M4). Public channel history receives channel
  chat send only.
- Townhall channel messages and agent-panel direct history remain separate
  presentation surfaces over the shared conversation store.
- A second in-flight send to the same panel is silently dropped. A missing
  panel or empty identifier also returns without a structured execution
  outcome.
- Closing an agent panel removes UI state but does not cancel or stop an
  in-flight request.
- Cancellation is currently returned as generic failure; admission, rejection,
  cancellation request, cancellation acknowledgement, timeout, disconnect,
  and terminal cancellation are not separate outcomes.
- Current agent results have no run/correlation ID, usage, finish reason,
  provider/model identity, or structured protocol failure. No object owns a
  cancellable agent run after dispatch.
- Identity is triplicated: Townhall seeds `user-1`/`agent-1`, agent panels seed
  `alpha`/`beta`/`gamma`/`delta` plus fallback names, and execution uses one
  global provider configuration with no agent identity. `WorkspaceAgent`,
  `AgentPanelState`, and execution configuration are not one registry.
- `TownhallMessage` reserves `SourceProvider`, `SourceModel`, `ThreadId`, and
  `Metadata`; `TownhallMessageKind` reserves `ToolCall` and `ToolResult`. Current
  producers do not form a typed event protocol around those fields and kinds.

### 4.2 Object ownership baseline

The editor provides the strongest existing object-ownership precedent:

```text
Workspace
  -> owns the open Document registry and ActiveDocument
Document
  -> owns content, dirty state, save-error state, and related events
```

Settings appropriately use immutable records and a stateful persistence
service. Workflow, LSP, and DAP use service-owned lifecycle state with
immutable snapshots and generation checks; this has been effective for V2 but
has also produced several large orchestration services.

Agent and Townhall state is comparatively anemic: behavior and ownership are
spread across state bags, ViewModels, coordinators, routers, the shell, and UI
string conventions. V3 must not turn `AgentPanelState`, `TownhallViewModel`, or
`MainWindowViewModel` into permanent all-purpose agent-domain owners.

### 4.3 Source and dependency baseline

The current production tree is organized almost entirely by technical layer:

```text
src/
  Models/       22 C# files
  Services/    224 C# files
  ViewModels/   53 C# files
  Views/        52 C# files (+ one AXAML dialog under Views)
  Styles/        2 C# files (+ Icons.axaml)
```

The application also has root `App.axaml` and `MainWindow.axaml` files; the
tree above counts only the named technical-layer directories.

Additional live findings:

- there is one production project, `src/Zaide.csproj`;
- `Zaide.Services` contains protocols, DTOs, interfaces, policies, parsers,
  snapshots, process owners, infrastructure, and application coordination;
- documented layering is not enforced by assembly or architecture tests;
- known `Services -> ViewModels` references include terminal session creation,
  mention parsing, and source-control diff-tab coordination;
- `SourceControlState` in `Models` consumes a service-layer snapshot;
- `Program.ConfigureServices` is one global registration method;
- `MainWindowViewModel` has a broad constructor and acts as a large feature
  composition surface;
- `MainWindow.axaml.cs` remains a large imperative shell-construction file;
- a V2-closeout text scan estimated roughly 339 public production types versus
  42 internal types; these are approximate declaration counts and Refactor 6.1
  M0 must remeasure them with a documented counting rule;
- DI registration is approximately 64 singletons, one transient, and no
  explicit scopes;
- static `App.Services` and direct `IServiceProvider` use hide dependencies in
  several composition paths;
- application exit manually resolves and disposes services in a fixed order;
- many state, identity, role, provider, and routing concepts remain raw
  strings;
- tests mirror the flat technical-layer organization, and multiple tests still
  carry phase/milestone names rather than durable product-behavior names.

These facts justify post-V2 structural refactoring. Before Phase 10, broad
source movement was correctly deferred because the active product phase had a
clear next milestone and no proven boundary failure justified the churn. V2 is
now closed, and concrete dependency, lifetime, composition, and ownership
problems have been identified.

## 5. Unified Conversation Model

V3 should treat channels and direct conversations as forms of one conversation
concept rather than as unrelated UI surfaces.

```text
Townhall
  |- Channels
  |    |- #general
  |    |- #project
  |    `- #agent-activity
  `- Direct Conversations
       |- Human <-> Agent
       |- Agent <-> Agent
       `- future Human <-> Human
```

Messages and execution events must be written to a stable `ConversationId`,
not to the currently selected UI item. Selection is presentation state;
conversation membership and message ownership are domain state.

The current dedicated agent-panel surface may remain as a temporary projection
during migration. It should be removed only after the Townhall direct-message
surface proves required retirement parity for:

- existing direct-send behavior;
- response and error visibility;
- stable conversation ownership;
- retained draft behavior;
- keyboard and focus behavior;
- no lost or silently detached in-flight state.

Phase 14 may add unified navigation, per-conversation draft/read state, and
structured event projection. Streaming, raw model I/O, backend reconnection,
tool events, and richer session resume are capability-dependent later behavior;
the conversation UI must be able to render them when later backends expose
them, but Phase 14 must not fake those capabilities merely to retire the old
panel.

Private detailed activity and public task summaries are different products.
Tool details, raw traces, and direct-message text remain in their owning direct
conversation unless an explicit sharing action and policy choose a public
target.

## 6. Identity, Participation, and Session Ontology

V3 must distinguish identity, runtime, session, conversation, and memory.

### 6.1 Recommended definitions

- **Actor** — a stable logical identity. Actor kinds may include Human, Agent,
  and System.
- **Agent Identity** — a durable Actor of kind Agent. It remains stable across
  profile revisions, runtimes, sessions, and backend changes.
- **Agent Profile** — a versioned persona and presentation/configuration record
  for an agent actor, including role, instructions, display configuration, and
  default backend preference.
- **Agent Runtime** — a live binding to one native-harness or external-agent
  backend and its negotiated capabilities.
- **Agent Session** — one resumable conversational/runtime context belonging to
  an Agent Identity. One agent may own many sessions.
- **Conversation** — an ordered communication space with a stable ID and a set
  of participants. Channel and direct conversation are conversation kinds.
- **Membership** — a durable association and role for an actor in a workspace
  or conversation.
- **Presence** — ephemeral connectivity/availability derived from runtime and
  session state; removing a runtime does not erase identity or authorship.
- **Policy Assignment** — memory, context, and permission policies attached at
  application, project, actor, conversation, or session scope.
- **Execution Run** — one bounded attempt to complete a user or agent task.
- **Tool Invocation** — one proposed, approved/rejected, executed, and recorded
  tool operation inside an execution run.

### 6.2 Identity decision

One session is not one agent. Sharing a memory file also does not make two
runtimes the same agent. The recommended identity rule is:

> One agent is one durable Agent Identity; sessions, runtimes, and memory scopes
> are resources associated with that identity.

This decision remains subject to evidence during Refactor 7 M0, but V3 plans
must not collapse identity into panel, process, provider, model, or session ID.

Candidate cardinalities for Refactor 7 M0 to verify are:

```text
Actor 1 -> many Memberships
Agent Identity 1 -> many Agent Profiles over time
Agent Identity 1 -> many Sessions
Conversation 1 -> many Memberships
Conversation 1 -> many Sessions and Runs over time
Runtime 1 -> zero or many Sessions
Session -> one Agent Identity + one backend binding + one workspace
```

The safest initial session rule is one session attached to one conversation;
Refactor 7 M0 must accept or reject it explicitly rather than leaving the
cardinality accidental. Historical events need an author/profile snapshot or
profile-version reference so renaming or archiving an actor does not rewrite
history.

### 6.3 Record taxonomy

These records have different truth, retention, and deletion semantics:

- **Conversation/event log** — authoritative communication and visible
  activity history.
- **Backend session state** — possibly opaque resumable backend context.
- **Run context manifest** — the exact selected inputs, versions, redactions,
  and budgets used for one run.
- **Durable memory record** — derived, editable knowledge.
- **Audit record** — security and action evidence.
- **Raw diagnostic trace** — optional sensitive debugging material.

Correcting or deleting memory must not silently rewrite conversation history
or audit evidence. Deleting a conversation also must not imply that opaque
backend state or provider-retained data was deleted unless that deletion was
confirmed.

## 7. Backend-Neutral Agent Experience

Townhall must consume a backend-neutral session and event contract while each
backend retains its own implementation.

```text
Townhall / Agent Direct Conversation
                 |
       Backend-neutral Agent Session
                 |
        +--------+--------+
        |                 |
Zaide Native Harness   ACP Backend
        |                 |
Provider API + tools   External agent runtime
```

The common application contract should eventually cover, where supported:

- start, resume, send, cancel, and end session;
- text and structured event streaming;
- tool/activity reporting;
- permission requests and decisions;
- file-change and command-result reporting;
- usage, cost, failure, and completion reporting;
- raw trace exposure;
- capability discovery and change notification.

Normalized execution outcomes must distinguish admission and runtime state:
accepted, queued, running, completed, failed, rejected, cancelled, timed out,
disconnected, and indeterminate. A cancellation request, backend
acknowledgement, and confirmed terminal cancellation are separate events. Late
completion after cancellation must remain representable rather than being
silently dropped.

Every normalized event needs, where applicable:

- event ID and schema version;
- backend source;
- conversation, session, and run IDs;
- optional tool/process ID;
- correlation, causation, and parent/delegation IDs;
- per-stream sequence/order key;
- occurred-at and received-at timestamps;
- evidence and visibility levels;
- payload type/version.

Replay rules must define ordering gaps, duplicates, idempotency, unknown
extension preservation, and valid terminal-event transitions.

The contract must not force all backends into a dishonest lowest common
denominator. Capability negotiation should describe features such as file
editing, command execution, permission interposition, session resume,
attachments, structured diffs, raw model I/O, and backend-specific extensions.

Equal treatment means equal placement, identity, conversation semantics, and
honest status presentation. It does not mean claiming capabilities that an
external backend does not expose.

Capability has multiple layers. Static runtime advertisement, session-specific
availability, current configuration, granted permission, and temporary
degradation are different states. `Advertised`, `available`, `permitted`, and
`currently usable` must not be collapsed into one Boolean. Capability changes
during a session need deterministic versioning and UI projection; backend-
specific extensions should be namespaced and preserved.

Zaide must also distinguish evidence levels for activity:

| Evidence level | Meaning |
|----------------|---------|
| Zaide-executed | Zaide performs the operation and can enforce policy |
| Zaide-mediated | The backend requests the operation through a Zaide-controlled surface |
| Backend-executed and reported | Zaide receives a claim but did not execute the operation |
| Externally observed | Zaide detects resulting state without reliable action attribution |
| Unobservable | Zaide cannot inspect or verify the operation |

A capability claim is not permission, and a reported event is not proof that
the action matches workspace reality. UI, audit, and trust language must retain
that distinction.

## 8. Zaide Native Harness

The native harness is a first-party Zaide subsystem. It should be built from
measured evidence gathered from multiple open-source coding-agent systems,
including candidates such as Qwen Code and OpenCode after their current source,
license, architecture, and runnable behavior are verified.

### 8.1 Research obligations

Before production harness architecture is locked:

- inventory several relevant open-source harnesses at exact versions/commits;
- verify their licenses and transitive obligations;
- trace their task loop, context selection, file search, editing, tool
  execution, failure recovery, and compaction behavior;
- run comparable repository tasks rather than judging architecture only from
  source appearance;
- distinguish copied code, modified code, translated code, and ideas only;
- record rejected approaches and the evidence for rejection;
- avoid benchmark overfitting by using varied task classes and held-out tasks.

### 8.2 Evaluation dimensions

The native harness should be evaluated on:

- task success and final test pass rate;
- regression and unnecessary-change rate;
- edit precision and final diff quality;
- observable model turns, searches, tool invocations, retries, and edit/test
  cycles;
- recovery after tool, command, or test failure;
- token, time, and monetary cost;
- context relevance and compaction quality;
- cancellation and process cleanup;
- permission correctness;
- reproducibility across models and repositories.

The goal is not merely to own a harness. The goal is to produce the best
practical agent performance Zaide can justify with repeatable evidence.

Comparison must include a controlled track using the same model/version,
provider settings, repository snapshot, tool availability, prompt policy,
resource budget, and acceptance tests where technically possible. Comparisons
that cannot control those variables are observational, not causal. Repeated
trials should report variance and catastrophic failures, reset the repository
and environment per attempt, include failed-attempt cost, use rubric-based or
blinded diff review where practical, track benchmark licensing/contamination,
retain held-out repositories beyond Zaide, and preserve reproducible result
artifacts.

### 8.3 Open-source provenance policy

Copied or adapted code must carry a provenance record with:

- upstream project and repository URL;
- exact commit or release;
- source file and copied range or component;
- upstream license and compatibility decision;
- required copyright and NOTICE text;
- local modifications;
- date introduced and responsible Zaide component;
- update/security tracking decision.

The record must cover more than copied C# source: dependencies, vendored or
modified source, binaries, generated/translated code, assets, prompts,
documentation, benchmark fixtures, and test corpora may all carry obligations.
Record the complete applicable license set, SPDX identifiers where applicable,
NOTICE/source-offer/patent/trademark obligations, preserved upstream notices,
prominent modification notices, SBOM or lockfile entry, and the final adopt,
isolate, rewrite, or reject decision. Release checks must verify that required
licenses and notices ship with distributed artifacts.

License verification must happen before code is copied into production. V3 may
reuse good implementation work; it may not erase its origin.

## 9. ACP as an Independent Backend

In this roadmap, ACP means the **Agent Client Protocol**. Its implementation M0
must use live primary documentation to pin an exact stable protocol version,
SDK version, transport profile, and supported extension set. Proposed or draft
protocol behavior must not be treated as stable. Every desired Zaide
capability must be classified as required protocol behavior, optional protocol
behavior, Zaide extension behavior, or unsupported behavior.

ACP exists to preserve user choice when the native harness is not preferred or
when an external agent is accessed through a process, account, subscription,
or other non-provider-API form.

ACP integration must therefore:

- remain independent of native-harness internals;
- connect through the backend-neutral application/session boundary;
- expose capabilities truthfully;
- preserve equal Townhall placement and conversation semantics;
- support permission and activity mediation where the external backend permits
  it;
- report where Zaide cannot observe or interpose an external action;
- handle disconnect, reconnect, cancellation, process exit, protocol errors,
  and capability changes;
- avoid requiring Zaide to expose a public commercial API.

Equal placement does not permit automatic identity inheritance. An external
runtime binds to an Agent Identity only through an explicit, verifiable
association. Reconnection must not silently attach a different external
runtime to an existing identity.

An ACP backend may expose protocol frames without exposing the external
agent's internal model prompt or response. Raw-trace UI must label that
boundary honestly.

Protocol compatibility also does not grant entitlement to an external account,
subscription, model, login, or non-API access method. ACP adapters must follow
the external agent's supported authentication, licensing, and service terms.

## 10. Live IDE Context

Agents need timely IDE state without receiving an unbounded event flood.

Potential context sources include:

- active project and selected project file;
- open and active documents;
- dirty buffers and selected text;
- diagnostics;
- build, run, and test lifecycle/results;
- debug start, stop, continue, exception, breakpoint, selected stack frame, and
  selected variables;
- terminal or process outcomes;
- source-control status and agent-attributed changes.

The context feed should use structured snapshots/deltas, stable correlation
IDs, rate limits or coalescing, privacy filtering, and explicit provenance.
Raw terminal output, every debug event, or complete variable trees should not
be included by default.

Context is assembled per run and recipient, not broadcast to every participant
merely because they share a conversation. Each run context manifest should
record source, scope, version/fingerprint, redaction state, destination trust
boundary, and token budget. Manual pin/attach is a first-class complement to
automatic selection.

### 10.1 User-configurable context policy

A likely starting model is:

| Level | Meaning |
|-------|---------|
| Off | No automatic IDE context sharing |
| Minimal | Failures, exceptions, stops, and essential task state |
| Standard | Active file, relevant diagnostics, and normal execution state |
| Detailed | Stack, selected variables, richer activity, and broader context |
| Custom | Event/source-level user configuration |

Defaults must be tuned through implementation evidence. Users should be able
to override the policy by application, project, agent, and session where that
complexity proves usable.

A candidate policy precedence is:

```text
Session override
  > Agent policy assignment
  > Project policy
  > Application default
```

Sensitive debug variables, terminal content, and dirty buffers remain excluded
unless explicitly selected or allowed. Automatic context sharing, tool-access
permission, and transmission to an external provider are separate controls.
`Off` disables automatic injection; it does not silently grant or deny explicit
file/tool access.

## 11. Raw Trace and Transparency

“Raw trace” means the deepest backend-exposed payload available **after
mandatory safety processing**. It does not promise private hidden reasoning,
chain-of-thought, or data an external backend does not expose. User-facing
reasoning summaries are ordinary structured output, not hidden-thought
disclosure. Enabling trace viewing must not change what context is sent to a
provider.

Normal conversation UI should render concise structured activity. A separate
optional raw trace should expose the deepest truthful level available:

- exact model request and response where the backend exposes them;
- selected context and memory injection;
- tool-call request, arguments, result, retry, and timing;
- permission evaluation and user decision;
- file and command operations;
- token, time, and cost usage;
- ACP protocol frames;
- errors, truncation, compaction, and fallback behavior.

Raw traces create their own security and reliability obligations:

- redact API keys, tokens, secrets, sensitive environment variables,
  configured file patterns, tool results, and sensitive file content **before
  persistence, indexing, rendering, export, logging, or telemetry**;
- define retention, size limits, rotation, export, and deletion;
- mark capture as disabled, unavailable, redacted, sampled, truncated, or
  summarized where applicable;
- separate capture/retention policy from display verbosity;
- prevent trace rendering from blocking the agent event pipeline;
- apply backpressure and avoid unbounded in-memory collections.

Durable audit evidence and optional sensitive diagnostic traces are different
record classes. Security audit must not depend on a user enabling raw capture,
and raw diagnostic retention must not become permanent by accident.

## 12. Memory and Context Quality

V3 should distinguish at least these memory scopes:

| Scope | Purpose |
|-------|---------|
| Session | Current task and resumable conversation context |
| Agent | Durable memory associated with one Agent Identity |
| Shared | Project/team knowledge available to multiple actors |
| Conversation | Channel or direct-conversation history and derived context |

Storage and prompt injection are separate concerns. Persisting memory does not
mean inserting it wholesale into every model request.

Memory retrieval should consider:

- relevance;
- provenance and author;
- creation and last-validation time;
- project/workspace scope;
- conflicts and superseding facts;
- token budget;
- summary versus original evidence;
- user visibility and control.

Users need the ability to inspect, correct, delete, disable, and scope durable
memory. The system should report which memories influenced a run where
practical. V3 must investigate memory poisoning, stale-memory amplification,
privacy, deletion propagation, and context-quality regression.

## 13. Tools, Permissions, and Audit

Native and external backends should enter a common Zaide control plane where
Zaide can truthfully mediate the action.

```text
Agent proposes action
  -> capability and policy evaluation
  -> allow / deny / ask user
  -> execute or delegate
  -> record result
  -> attribute changes to run and actor
```

Permission dimensions may include:

- read versus write;
- workspace-internal versus external paths;
- process and terminal commands;
- network access;
- Git reads and mutations;
- secret access;
- destructive operations;
- memory read/write scope;
- one-time, session, project, or persistent approval.

V3 must address approval race conditions: the operation actually executed must
match the operation or diff the user approved. External-agent actions that
Zaide cannot interpose must be labeled as such rather than presented as fully
sandboxed.

Approval should bind to a canonical action description, including where
applicable:

- executable and argument vector rather than rendered shell text alone;
- resolved working directory;
- normalized paths and resolved real paths;
- relevant file revisions or diff hash;
- environment additions and network destination;
- capability, actor, run, expiry, and approval scope.

Any material mutation requires reevaluation. Unknown or unverifiable action
shapes default to deny or ask, never implicit allow. Policy precedence,
revocation, and the distinction between trusting a workspace and granting a
specific action remain explicit.

The audit model should correlate:

```text
Conversation
  -> Agent Session
    -> Execution Run
      -> Tool Invocation
        -> Process / File Mutation / Result
```

### 13.1 Agent process ownership

Each runtime, session, run, and child process must have one explicit lifecycle
owner. Native tool processes and locally launched external-agent processes need
contracts for:

- executable origin, version, and trust decision;
- argument, working-directory, and environment construction without shell
  interpolation;
- minimal environment and secret inheritance;
- initialization, handshake, and authentication timeout;
- stdout, stderr, and protocol-stream separation;
- bounded output and event queues;
- cooperative cancellation followed by bounded forced process-tree
  termination;
- workspace close, session close, app shutdown, crash, and orphan recovery;
- reconnect rules that cannot duplicate a still-running operation;
- budgets for process count, descendants, CPU, memory, disk, and output.

Workspace close and application shutdown must not leave unowned child
processes. Crash recovery must not trust a reused PID as proof of process
identity.

## 14. Workspace Mutation and Concurrency

Real agent editing introduces conflicts that ordinary chat does not.

V3 must define:

- whether open dirty editor buffers or on-disk files are authoritative for
  each operation;
- how an agent reads unsaved user edits;
- how agent edits appear in open documents without silent overwrite;
- optimistic concurrency/version checks;
- conflicts between multiple agents editing the same file;
- conflicts between build/test/debug and file mutation;
- an agent-attributed change set and reviewable diff;
- rollback that removes an agent's changes without destroying unrelated user
  or agent work;
- file-watcher and external-edit reconciliation;
- cancellation semantics for partially applied multi-file edits.

The native harness must not bypass the V2 `Workspace`/`Document` ownership
model by treating every edit as an uncoordinated disk patch. ACP backends may
have different edit mechanisms; the capability and trust model must state what
Zaide can observe and reconcile.

## 15. Security and Trust Questions

V3 requires an explicit threat model before autonomous tool execution.

At minimum, discovery must cover:

- prompt injection from repository files, command output, diagnostics, web
  content, memory, and other agents;
- accidental or malicious secret exfiltration;
- workspace trust on first open;
- path traversal, symlink escape, and workspace-boundary enforcement;
- command substitution and shell interpretation;
- untrusted ACP processes and false capability/activity reporting;
- dependency and copied-code supply-chain risk;
- raw-trace disclosure;
- memory poisoning and cross-project leakage;
- user impersonation and actor identity integrity;
- denial of service through tokens, processes, events, files, or recursion;
- Agent-to-Agent loops, runaway delegation, and deadlock.

Security policy should remain observable and understandable. A hidden
permission engine is not sufficient for an agent-first IDE.

## 18. First V3 Feature Direction

After the required refactors close, the first expected feature phase is:

### Phase 14 — Unified Conversation Workspace

**Candidate outcome:** Public Townhall channels and Agent direct conversations
share one conversation system, and the dedicated Agent Panel can be retired
after parity is proven.

Expected concerns include:

- channel and direct-conversation navigation;
- stable membership and unread/draft state;
- Agent DM send, cancel, retry, failure, and execution-status visibility;
- structured conversation/event rendering;
- migration from the temporary agent-panel projection;
- keyboard, focus, and visual-design evidence;
- honest backend capability presentation.

Phase 14 M0 must also lock the persistence and recovery contract needed to
retire the dedicated Agent Panel:

- persisted versus in-memory conversation history, drafts, unread/read cursor,
  membership, and active-conversation state;
- how interrupted runs appear after restart or backend disconnect;
- a rule that side-effecting work is never blindly resumed after restart;
- ordering, idempotency, deduplication, and replay behavior;
- workspace and window isolation;
- migration and rollback for any V2 conversation state that is retained.

The UI acceptance boundary should include incremental or virtualized rendering
for growing conversations, stable scroll anchoring, near-bottom auto-follow,
and a new-message affordance when the user has scrolled away. It should also
include semantic controls, keyboard-only navigation, visible focus, screen-
reader naming, and narrow/wide/high-DPI evidence. Phase 14 needs a short design
brief and comparison screenshots against `docs/DESIGN.md`; the current
Discord-like composition is reference scaffolding, not the visual target.

Phase 14 scope is not locked until its own M0. Native Harness production
implementation and ACP integration are later V3 feature concerns unless Phase
14 M0 proves that a narrower backend seam is required to deliver truthful DM
behavior.

### Phase 15 — Backend-Neutral Agent Session and Event Foundation

**Accepted M0 outcome:** Establish a backend-neutral, in-memory Agent Session
and structured event foundation over the unified conversation model, then adapt
the existing non-streaming OpenAI-compatible execution path as the first
concrete compatibility backend.

The live dependency order requires this foundation before either backend path:

```text
Phase 14 unified conversations
  -> Phase 15 backend-neutral session/run/event foundation
      -> later Zaide Native Harness backend
      -> later independent ACP backend
```

Phase 15 is intentionally narrower than Native Harness or ACP implementation.
It may define session identity/lifecycle, run states, ordered structured events,
capability snapshot/versioning, evidence levels, and projection into existing
conversation entries. It must preserve Phase 14 privacy, persistence, and
no-auto-resume behavior.

Phase 15 does **not** implement tools, permissions, workspace mutation, usage or
cost reporting, raw trace, session persistence/resume, the Native Harness, or
ACP. Capability, availability, configuration, permission, degradation, and
current usability remain distinct facts. Backend-reported activity is not
Zaide-verified execution.

## 19. Additional Discovery Questions

The following remain open and must be assigned to later discovery/refactor M0
work rather than answered by assumption:

### Persistence and compatibility

- Which store owns conversations, sessions, audit events, raw traces, and
  memory?
- What are the retention, export, delete, backup, and migration contracts?
- How are event ordering, idempotency, replay, and schema versioning handled?
- How is cross-workspace or multi-window state isolated?
- Which restart state is recoverable, and which interrupted operations must be
  presented as terminal or indeterminate instead of resumed?

### Runtime and scheduling

- How many agents and runs may execute concurrently?
- How are priority, budget, rate limits, recursion, and Agent-to-Agent loops
  controlled?
- How are token, time, cost, process, disk, and event budgets surfaced?
- What survives application crash or backend disconnect?

### Trust and capability

- How does a workspace become trusted for agent execution?
- How are backend capability changes represented during a session?
- What evidence can Zaide independently verify for an external agent's edits
  and commands?
- What fallback behavior is allowed when a backend cannot support a requested
  permission or trace feature?
- Which exact stable ACP protocol/SDK version, transport profile, and optional
  extensions will Zaide support? Verify these from primary sources during the
  implementation M0 rather than relying on this discovery snapshot.
- Which exact repositories, commits, licenses, NOTICE obligations,
  dependencies, prompts, assets, test corpora, and generated artifacts belong
  to each Qwen Code/OpenCode candidate? Project-name ambiguity must be resolved
  before evaluation or reuse.
- Which named external agents actually support the selected ACP profile, and
  what authentication or service terms govern their use? Protocol
  compatibility alone is not entitlement to a subscription or login flow.

### Product and design

- How are humans and agents shown together without copying Discord's visual
  identity?
- Which activity is inline, collapsed, summarized, or moved to raw trace?
- How do users understand who is present, busy, disconnected, or delegated?
- How should notifications and unread state avoid overwhelming the user?
- How will eventual Human-to-Human messaging fit without being implemented
  prematurely?
- Who may create, connect, disconnect, archive, remove, or rename an agent or
  conversation membership, and what happens to historical authorship after
  removal?
- How are duplicate display names handled while stable IDs remain the routing
  authority?
- How quickly do presence and policy revocation propagate, and which stale
  states remain visible after reconnect?

Future Human-to-Human support must not require a second conversation model.
Actor kind should remain neutral, direct conversations should be participant
sets rather than hard-coded “user plus one agent” records, display names must
never be routing keys, and authentication origin must remain extensible. These
are compatibility invariants, not authorization to build multi-user messaging
in the initial V3 sequence.

### Evaluation

- What reproducible task corpus compares native-harness designs?
- Which tasks remain held out?
- How are model variance, provider changes, and environmental failures
  separated from harness performance?
- How are copied upstream components tracked and re-evaluated?
