# Phase 15 M1a Evaluation Protocol

## Status and boundary

This is an implementation-ready specification for a future controlled
comparison. It does not implement a runner, create fixtures, execute a model,
install a candidate, rank a harness, or authorize adoption.

Authorized research candidates are Qwen Code, OpenCode, and Grok Build. Their
presence in this protocol means only that a later, separately authorized
campaign could instantiate the same manifest for each. M1a executed no trial.

Verification/specification date: **2026-07-21**.

## Comparison unit and labels

A comparison unit is one immutable tuple:

```text
candidate snapshot
+ candidate configuration
+ model/provider snapshot
+ repository/task snapshot
+ prompt/context/tool/permission/network policy
+ resource budget
+ trial number
+ acceptance rubric version
```

Changing any tuple field creates a different configuration. Results from
different configurations may be reported together only as explicitly
stratified observations.

Every conclusion must carry one label:

- **Causal:** all predeclared comparison controls matched, reset proof passed,
  candidate identity was the intended independent variable, and no material
  post-hoc exclusion occurred.
- **Observational:** one or more controls could not be matched or verified.
- **Invalid:** reset, artifact, integrity, or acceptance evidence is missing;
  do not use the trial in comparative statistics.

Unknown control state is observational or invalid, never causal.

## Controlled comparison manifest

The future runner must serialize one manifest before each trial. Fields marked
`required` cannot be inferred after execution.

| Section | Required fields |
|---------|-----------------|
| Manifest identity | schema version; experiment ID; trial ID; UTC creation time; operator/automation identity; rubric version; task version |
| Candidate | name; repository URL; exact repository commit; default branch observed; release/tag; tag-ref commit; release metadata target; source publication date; local patches with hashes; binary version/hash/source mapping; `SOURCE_REV` or equivalent when present |
| Model/provider | provider; endpoint class; exact model ID/version; API/protocol; temperature; top-p; seed; reasoning/effort; max output; retry policy; service tier; provider-side tools; server configuration identifiers where exposed |
| Repository | repository URL; exact commit; submodules; LFS state; clean/dirty status; sparse-checkout state; case sensitivity; line endings; filesystem/worktree path; license |
| Environment | OS/kernel; architecture; CPU/RAM; container/VM; locale/timezone; shell; runtime/compiler/package-manager versions; candidate version output; environment-variable allowlist with secret values redacted |
| Prompt/context | exact system/developer/task prompts by content hash; instruction-discovery policy; supplied files; hidden context; memory/cache state; conversation history; compaction policy; maximum context |
| Tools/permissions/network | exact tool names/versions/schemas; permission rules; sandbox profile; workspace boundary; writable paths; command allow/deny rules; network deny/allow policy; MCP/plugins/hooks/skills/subagents enabled state |
| Budgets | wall-clock; model turns; input/output/reasoning/cached tokens; monetary cost; tool calls; command invocations; retries; generated bytes; maximum diff files/lines |
| Acceptance | automated command list; expected exit codes; rubric criteria/weights; forbidden changes; allowed flaky tests declared before trials; reviewer blinding method |
| Artifacts | artifact directory; required filenames; hashing algorithm; redaction policy; truncation policy; encryption/access class; retention deadline; deletion status |
| Reset proof | pre-trial and post-trial repository hashes/status; process/network cleanup; cache policy; dependency state; workspace snapshot ID; previous-trial artifact absence |
| Classification | causal/observational/invalid; unmatched controls; deviations; exclusion status and reason |

Secret values must never be stored in the manifest. Store only the variable
name, source class, presence, and a non-reversible fingerprint when correlation
is necessary and approved.

## Candidate identity rules

### Qwen Code

Record repository HEAD and the stable release/tag independently. A preview or
nightly tag is a different candidate configuration. Do not infer that an npm
package or installed binary exactly matches a source commit without published
provenance and a verified artifact hash.

### OpenCode

Record the default-branch commit, release metadata target, tag-ref commit, and
binary/package hash separately. A release metadata target that differs from
the tag ref is not normalized away.

### Grok Build

Record all of these independently:

1. public repository commit;
2. monorepo commit in `SOURCE_REV`;
3. GitHub release/tag, explicitly `none` when absent;
4. distributed binary version and hash, if a future trial is authorized;
5. changelog entry/version; and
6. service/model identifiers.

No Grok Build trial is comparable as a source-build trial unless the public
source builds the exact tested artifact under a recorded reproducible process.
No binary trial may claim public-source equivalence without primary evidence.

## Environment and repository reset proof

Before every trial, the future runner must:

1. terminate candidate-owned child processes from the previous trial;
2. provision a fresh worktree or restore a content-addressed snapshot;
3. verify repository `HEAD`, submodules, LFS objects, file mode, and line-ending
   policy;
4. prove `git status --porcelain=v2 --untracked-files=all` is empty unless the
   task manifest declares seeded untracked files;
5. restore dependency directories, build outputs, caches, candidate state,
   conversation/session state, and credentials according to the frozen cache
   policy;
6. hash the task inputs and repository tree before execution;
7. record active processes, listening ports, mounts, writable paths, and
   network policy; and
8. fail the trial before model invocation if any reset assertion fails.

After the trial, capture the final tree hash, complete diff including untracked
files, active child processes, and cleanup outcome. Cleanup failure does not
erase the trial; it marks the trial invalid and remains in the artifact set.

## Controls

### Model/provider/configuration

- Use the same exact model/provider/endpoint/service tier for a causal harness
  comparison.
- Freeze all exposed sampling, reasoning, context-window, retry, and timeout
  options.
- Record provider-side prompt/tool transformations when exposed. If they are
  opaque or candidate-specific, label the comparison observational.
- Do not replace missing usage or cost with zero. Record `unavailable`.
- Do not silently retry a failed trial outside the candidate's frozen retry
  policy. A rerun is a new trial number.

### Prompt and context

- Byte-identical task, system, and developer prompts are required unless the
  protocol explicitly studies native prompt behavior.
- If native candidate system prompts remain enabled, the comparison is a
  whole-system comparison; record their hashes and do not claim the harness
  loop alone caused the result.
- Freeze instruction files, supplied context, repository-discovery boundary,
  memory state, skills, plugins, hooks, MCP servers, and subagent definitions.
- Record candidate-added context when observable. Opaque context makes the
  related claim observational.

### Tools and permissions

- Expose capability-equivalent tools with equivalent schemas, timeouts,
  output limits, writable boundaries, and permission rules for causal claims.
- Preserve native tools only in a separately labeled whole-system track.
- A denied or cancelled action remains part of the trial result.
- Record tool input, normalized result, exit code, stdout/stderr, duration,
  permission decision, and cancellation evidence, subject to redaction.

### Network

- Default evaluation fixtures to no network after dependency provisioning.
- Any required network destination must be allowlisted by hostname/service and
  recorded with request timing and failure evidence where possible.
- Provider traffic and task-required traffic are separate categories.
- Unrecorded network access invalidates a causal claim.

### Budgets

- Use identical wall-clock, turn, token, cost, tool-call, command, retry, and
  output budgets for causal comparison.
- A candidate that reaches a budget stops with a distinct `budget_exhausted`
  outcome; the trial is not discarded.
- Different budgets may be studied only as separate configurations and must
  not be presented as a harness-only comparison.

## Task classes

Use small, licensed, reproducible tasks across more than one repository:

1. locate and explain without mutation;
2. one-file behavioral bug with a focused test;
3. multi-file feature with explicit architecture constraints;
4. diagnose and recover from a failing command or test;
5. precise refactor with behavior preservation;
6. cancellation/interruption and child-process cleanup;
7. stale-context or conflict recovery; and
8. held-out tasks and held-out repositories unavailable during tuning.

Zaide may be one repository but must never be the only evaluation repository.
Every task records repository/corpus license, provenance, contamination risk,
and whether model training exposure is known, suspected, or unknown.

## Acceptance rubric and metrics

### Outcome rubric

Score each criterion independently; never replace test evidence with a single
subjective score.

| Criterion | 0 | 1 | 2 |
|-----------|---|---|---|
| Required behavior | Missing or wrong | Partial/ambiguous | All acceptance checks pass |
| Regression control | Material regression | Unresolved or incomplete evidence | Required focused and full gates pass |
| Scope discipline | Destructive/out-of-scope changes | Minor unrelated churn | Only necessary in-scope changes |
| Code quality | Unmaintainable/unsafe | Acceptable with issues | Clear, idiomatic, architecture-compliant |
| Recovery | Failure ignored or state corrupted | Failure surfaced without full recovery | Failure classified, recovered/terminated truthfully, cleanup proven |
| Safety/permissions | Boundary violation | Ambiguous evidence | Every action authorized and bounded |
| Reproducibility | Missing manifest/artifacts | Partial evidence | Complete reset and artifact proof |

Any destructive boundary violation, secret exposure, missing final diff, or
unproven reset is a hard failure regardless of aggregate score.

### Required measurements

| Dimension | Measurement |
|-----------|-------------|
| Success | Acceptance command pass/fail, rubric criteria, partial completion, terminal outcome |
| Regression | Focused/full-suite results and attributable behavior regressions |
| Diff quality | Necessary files/lines, unrelated churn, generated artifacts, architecture/convention violations, reviewer score |
| Recovery | Tool/model/command failures, retries, stale context, cancellation latency, cleanup, terminal classification |
| Efficiency | Wall time, model turns, searches, tool calls, command/edit/test cycles, latency distribution |
| Tokens/cost | Input/output/cached/reasoning tokens and complete/incomplete monetary cost; `unavailable` rather than zero |
| Safety | Permission decisions, workspace/network violations, destructive actions, secret/redaction events |
| Reproducibility | Reset proof, environment fingerprint, artifact completeness, repeated-trial variance |

Report raw values and distributions. Do not collapse all dimensions into one
leaderboard score during Phase 15.

## Repeated trials and variance

- Run at least three trials for every task/configuration before any comparative
  claim. Use more when the predeclared confidence/precision target is unmet.
- Predeclare trial order and randomization. Rotate candidate order to reduce
  time/provider-load bias.
- Report every trial, failure, timeout, cancellation, budget exhaustion, and
  catastrophic outlier.
- Report median, minimum, maximum, interquartile range, and success proportion.
  With small samples, emphasize raw trials and intervals rather than a fragile
  mean.
- Do not remove an outlier because it is inconvenient. Exclusion is allowed
  only for a predeclared integrity failure, and both the excluded trial and the
  reason remain visible.
- A provider incident or unmatched rate limit is an observational stratum, not
  a silent rerun.

## Held-out separation

- Keep tuning tasks/repositories and held-out tasks/repositories in different
  access-controlled manifests and artifact roots.
- Candidate configuration, prompt, tool, or rubric changes may use tuning data
  only.
- The first inspection of a held-out task permanently moves it out of held-out
  status for the inspecting team/configuration.
- Do not tune acceptance tests after seeing candidate output. If the task was
  ambiguous, invalidate it, version the task/rubric, and rerun all candidates.
- Report held-out results only after configuration lock.

## Artifact retention, hashing, redaction, and truncation

### Required artifacts

Retain for every trial:

- immutable manifest and reset proof;
- exact prompts and supplied context, or approved content-addressed references;
- candidate-visible event/transcript stream;
- tool calls, permission decisions, commands, exit codes, stdout/stderr;
- model/provider usage and cost fields exactly as exposed;
- final diff including untracked files;
- test and acceptance outputs;
- timing, process, network, and environment evidence;
- failure, retry, cancellation, and cleanup evidence; and
- rubric scoring with reviewer identity/blinding state.

### Integrity

- Use SHA-256 for every artifact and a canonical manifest-of-hashes.
- Sign or append the root manifest hash to an immutable experiment log.
- Hash the original artifact before redaction, store the original only in an
  approved restricted location, and hash the redacted derivative separately.
- Any mutation creates a new artifact version; never overwrite evidence.

### Redaction

- Redact secrets, credentials, tokens, personal data, private URLs, and
  prohibited repository content before general review.
- Record the redaction rule ID, field/path, reason, replacement marker, and
  original restricted hash. Never store the secret itself in the redaction
  log.
- If redaction removes evidence needed for acceptance, keep the decision
  restricted to authorized reviewers or mark the criterion unavailable.

### Truncation

- Prefer chunked full artifacts over truncation.
- When truncation is required, preserve head and tail, byte counts, omitted
  range/hash, encoding, and reason.
- UI-visible clipping is not an acceptable evidence artifact. Capture the raw
  stream/file when authorized.

### Retention

- The future campaign must predeclare retention by artifact class, storage
  location, access control, and deletion date.
- Provider-side retention is a separate external fact and must not be inferred
  from local artifact policy.
- Failed trials follow the same retention rules as successful trials.

## Minimal conceptual Phase 15 contract fixtures

These are specifications only. M1a creates no fixture files and runs no
campaign.

| Fixture ID | Minimal scripted observations | Phase 15 contract evidence |
|------------|-------------------------------|----------------------------|
| `P15-F1-single-completion` | One admitted user message; backend reports one non-streaming assistant completion; usage/raw trace omitted | One session/run/conversation/backend identity; monotonic event sequence; truthful message/completion; unavailable values remain unavailable |
| `P15-F2-admission-rejection` | Hold one run active; submit a second send to the same conversation/session | Explicit accepted/running state for the first run and structured rejection for the second; no fabricated execution/tool event |
| `P15-F3-cancellation-race` | Request cancellation while the backend is active; script either confirmed cancellation or late completion as separate variants | Cancellation request is distinct from terminal cancellation; late completion remains representable; event ordering/correlation is exact |
| `P15-F4-terminal-failure` | Script one transport failure and one timeout variant with no usage/cost/raw trace | Failed versus timed-out terminal states remain distinct when evidence exists; missing telemetry is not synthesized; conversation projection is exactly once |

These fixtures validate Zaide's session/event contract shape later. They are
not representative harness benchmarks and cannot support a candidate ranking.

## Analysis and reporting rules

1. Publish the manifest and raw per-trial table before aggregate claims.
2. Separate task success from efficiency, safety, recovery, and diff quality.
3. Separate whole-system effects from loop/tool architecture effects.
4. Mark unobservable provider/server behavior as unknown.
5. Describe association as observational unless all causal controls passed.
6. Never recommend adoption from this protocol alone; provenance, security,
   product fit, maintenance, and human review are independent gates.

## Rejected-approach log

| Rejected approach | Reason |
|-------------------|--------|
| Uncontrolled comparisons labeled causal | Candidate, model, provider, prompt, environment, or budget effects cannot be separated. |
| Different model, provider, context, tool, or cost budgets presented as harness comparisons | The harness is not the isolated variable. |
| Zaide-only evaluation | It overfits one architecture, language, test stack, and repository history. |
| Tuning on held-out tasks or repositories | It destroys held-out status and inflates reported generalization. |
| Discarding failures, timeouts, cancellations, or outliers | It hides reliability and recovery behavior and biases aggregates. |
| Broad harness ranking during Phase 15 | Phase 15 needs contract evidence, not a leaderboard or selection decision. |
| Adopting material based only on a repository root license | Root licensing does not clear dependencies, subtrees, assets, prompts, corpora, generated artifacts, ports, or binaries. |
| Treating a release metadata target and tag-ref commit as interchangeable | Published identifiers can differ and must remain separately auditable. |
| Mapping any Grok Build changelog or binary version to the public repository or `SOURCE_REV` without primary evidence | No GitHub release/tag or verified mapping to a public source commit, monorepo commit, distributed binary hash, or hosted service exists. |
| Assuming public Grok Build source equals all shipped clients or server-side components | Public snapshot, private monorepo, build pipeline, binaries, and services are separate evidence domains. |
| Inferring privacy, repository upload, retention, telemetry, authentication, or remote-service behavior solely from policy text or the open-source announcement | Static source and product statements do not prove runtime configuration, deployed behavior, or server processing. |
| Using only successful reruns after a candidate failure | This conditions results on success and erases recovery/reliability evidence. |
| Replacing unavailable usage/cost with zero | Zero is a measurement; unavailable is missing evidence. |

## M1a completion lock

- No runner exists.
- No fixture file exists.
- No benchmark or model invocation occurred.
- No candidate is ranked or selected.
- No upstream material is adopted.
- Any later execution needs separate authorization, a provenance-cleared
  candidate artifact, and a pre-registered manifest.
- M1b remains unauthorized.
